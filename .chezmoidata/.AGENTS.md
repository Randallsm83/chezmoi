# .chezmoidata/

Static data files. Chezmoi auto-loads every `*.yaml` here and merges them into a **single top-level template namespace** — there is no per-file prefix. Editing any file here drives most repo-wide behavior changes. See [root AGENTS.md](../AGENTS.md) for the bigger picture and full variable reference.

## Ownership matrix

| File | Top-level namespaces | Edit when |
|---|---|---|
| `theme.yaml` | `.theme.*`, `.theme_mappings.*` | Adding a theme, retuning a palette, mapping a new app to existing themes |
| `fonts.yaml` | `.fonts.*` | Switching primary font, tweaking Fira Code OpenType features |
| `ssh.yaml` | `.ssh.*` | Changing 1Password vault order, agent paths |
| `packages.yaml` | `.package_features.*`, `.package_mapping.*`, `.brew_bundle.*`, `.scoop_buckets`, `.scoop_bucket_overrides`, `.always_install.*`, `.remote_packages.<tier>`, `.claude_memory_projects` | Adding a tool, flipping a feature flag, changing brew/scoop sources, defining remote-tier packages |
| `dns.yaml` | `.vpn_dns_routes.*`, `.encrypted_dns.*`, `.browser_doh.*`, `.caddy_ca.*` | Adjusting VPN split-DNS, NRPT/resolver rules, browser DoH policy, Caddy root trust |
| `mcp.yaml` | `.mcp.servers.*` | Adding/removing an MCP server (Claude Code, Warp, Zed, opencode, VS Code all render from this map) |

## Override surface

Every value here is overridable in `~/.config/chezmoi/chezmoi.toml` (or per-machine `.chezmoi.local.toml`):

```toml
[data]
theme = "onedark"             # was tokyonight-storm

[data.package_features]
rust = false                  # disable a feature flag

[data.ssh]
onepassword_vaults = ["Personal"]   # narrow vault list

[data.mcp.servers.deepwiki]
disabled = true               # NB: shape depends on consumer
```

After editing, run `chezmoi data` to verify the merged namespace.

## Naming gotchas

- **`1password` feature flag** starts with a digit → invalid Go-template identifier. Access only via `{{ index .package_features "1password" }}`, NEVER `.package_features.1password`.
- **No `.chezmoidata.yaml`** — the old monolithic file was split per-domain. Adding a new top-level domain → create a new `*.yaml` file here, do NOT re-introduce the monolith.
- **Schema mirrors the consumer** — `mcp.yaml` matches Claude Code's `mcpServers` shape because it's the most permissive; translators in templates downgrade for stricter consumers.

## Adding a new top-level namespace

1. Create `<topic>.yaml` here with a single top-level key matching the namespace.
2. Reference as `{{ .<topic>.* }}` in any template.
3. Add the override example to this file's matrix.

No registration step. Chezmoi merges automatically on next `apply`.

## Anti-patterns

- DO NOT put secrets here — `op://` references belong in `.chezmoi.toml.tmpl` (`$secretsTpl`), surfaced as `.secrets.*`. See [SECRETS.md](../SECRETS.md).
- DO NOT put machine-specific values here — that's `chezmoi.local.toml`'s job. This directory is committed; the local file is not.
- DO NOT add a per-platform variant file (e.g. `packages.windows.yaml`). Platform branching belongs inside the YAML (manager keys like `package_mapping.<name>.darwin.cask`, `package_mapping.<name>.linux.<manager>`) or in templates.
